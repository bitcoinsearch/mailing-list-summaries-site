Summary:

The context revolves around the use of extended public keys in multisig arrangements and the privacy and security issues associated with it, particularly concerning BIP-45. One major issue is that using the same extended public key for all multi-sig activity can lead to a degradation of privacy and potential security risks due to increased public key reuse. Different suggestions are made to solve this issue, including lexicographically sorting public keys and using separate branches for each cosigner.The email thread also discusses the team's approach to building Copay, a multisignature P2SH HD wallet, and outlines their general address generation procedure. Additionally, the email highlights the importance of not compromising privacy for convenience and suggests including such concerns in the BIP-45 document.A multisig account is an agreement between a set of co-signers to be signatories of multi-signature transactions that require a set number of signatures. Using the same set of extended public keys in multiple wallets is not recommended by BIP45. By adopting a scheme as an improvement, co-signers would have to know each other's extended public key to eavesdrop on any transaction between them.The email exchange took place through BitPay.com, and the developer reviewing BIP-45 suggests amending it. However, they are informed that BIP's are not amended after the fact, and it may be best to write a new BIP specifying a "pseudorandom & deterministic path generation for HD/multi-signature accounts." Jean-Pierre proposes a solution that involves taking all public keys encoded in the purpose-specific extended public keys (m/45') of all co-signers and sorting them lexicographically according to BIP-45. Serialize this information and calculate its HASH160 (RIPEMD160 âˆ˜ HASH256). Split the output into five 32-bit chunks, setting the MSB on all of them to 0. Use these 32-bit chunks to build a derivation path from the purpose-specific extended public keys. This scheme will avoid public key sharing, and as long as you share your purpose-specific extended public key only with your co-signers, it should be relatively hard for a passive observer to link activity between different co-signing accounts.The email also discusses the use of m/i'/45' as a multisig "account" number in a BIP45 wallet. It simplifies the workflow for establishing new multi-signature accounts significantly, and you can give every cosigner only that single piece of information, that single xpub. However, adding a new dimension to it will make things harder. If all wallet information is lost except for the mnemonic and extended public keys from all people established multi-signature accounts with, it becomes harder to restore the wallet from the initial mnemonic if something goes wrong.Finally, Tamas Blummer notes that unless users are expected to establish a single co-signing account, this scheme will result in reuse of public keys and degradation of privacy. He suggests that privacy and potential security degradation due to increased public key reuse in the case of users with multiple multi-signature accounts should get a mention in the BIP-45 document.
