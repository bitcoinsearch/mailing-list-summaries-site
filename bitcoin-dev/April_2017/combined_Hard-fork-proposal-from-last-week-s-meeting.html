Summary:

The Bitcoin community is currently engaged in a debate about scaling the network to handle increasing demand without compromising security or decentralization. One key issue is the block size limit, with some advocating for increases while others suggest alternative solutions like layer two scaling. Full nodes are seen as essential to maintaining monetary sovereignty and preventing miners from rewriting any of Bitcoin's rules. However, the cost of running nodes is a significant factor, with estimates suggesting it costs around $250k per year to run a node capable of handling global transaction volume. Various proposals have been suggested for addressing the block size issue, including hard forks and soft forks, but there are concerns about risks and impact on smaller miners. Wang Chun's proposal to remove the 1MB block size limit in Bitcoin Core's next release but not activate it until spring 2020 has been put forward. This patch would allow three years for third-party services, wallets, and exchanges to prepare for the fork. While some members of the community have suggested waiting until SegWit activates to truly measure the effects on the network from increased capacity before committing to any additional increases, others propose implementing a soft fork to a 32MB limit to end debates around block size.There are also concerns about the potential loss of decentralization and coordination issues with small blocks if a decision is made solely by Core. Some members of the community have suggested adding a runtime option to allow node operators to increase the block size limit, making it a less controversial decision. Before increasing the block size, several things need to be done, including defining a network protocol for UTXO snapshots and implementing SPV support from a pruned node with the latest snapshot. The importance of full nodes in arbitrating consensus and the cost of running them are discussed in this context. Home users may find it challenging to run a full node due to bandwidth caps, high disk space costs, and limited RAM. It is crucial to consider the impact of running Bitcoin on affordable, non-dedicated home-hardware and to periodically commit the entire UTXO set to a special checkpoint block to reduce bandwidth and disk pressure.Lightning potentially offers a solution that could scale Bitcoin significantly, making block size a non-issue for years to come. The importance of retaining power for users is also highlighted, with the ability to run a node keeping miners honest and protecting against changes to Bitcoin's rules. While some argue that low node costs are important to maintain a decentralized network, others believe that keeping block sizes small has significant costs for everyone. A proposal to create a true fee-market has been suggested, where miners can choose to make a block smaller to encourage higher fees. However, the risks associated with hard forks and the need for proper preparation before deployment are being debated by Bitcoin developers.
