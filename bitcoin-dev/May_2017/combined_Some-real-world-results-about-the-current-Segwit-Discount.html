Summary:

Several discussions have taken place on the bitcoin-dev mailing list regarding the implementation of SegWit in Bitcoin. The SegWit softfork discount is aimed at reducing UTXO bloat and minimizing witness spam. The current 75% discount was chosen for multisignature transactions to help Bitcoin scale without a hard fork, but Sergio Demian Lerner suggests that it only makes worst-case block sizes worse, increasing them to 4MB when they should be 2.7MB. He encourages a 50% witness discount instead to make scaling less risky by keeping the worst-case block size at 5.4MB instead of 8MB.Lerner processed 1000 blocks and found that the non-witness data weight factor should be 2.35 instead of 4. The current 75% discount can only achieve more transactions per second if the type of transactions change. However, this analysis doesn't take into account that LN transactions may affect the current state, increasing the segwit/nosegwit ratio. Johnson Lau notes that any parameter can be changed in a hardfork, so this discussion does not necessarily relate to the current BIP141 proposal.In response to Lerner's suggestion, Alphonse Pace disagreed with Sergio's suggestion, stating that a 75% discount was better for scalability without a hard fork. Matt Corallo also disagreed, emphasizing the importance of Segwit as a key feature for Bitcoin's long-term reliability. Gregory Maxwell chimed in with some playful banter.Based on a conversation between Sergio Demian Lerner and an unknown person, the capacity of Segwit with a 75% discount is calculated to be 701K non-segwit bytes, which translates to an average block size of 1.8MB and a maximum block size of 4MB. For Segwit with a 50% discount and a 2MB hard fork, the capacity is calculated to be 1.08M non-segwit bytes, which translates to an average block size of 2.9MB and a maximum block size of 4MB.In another thread, they discussed limiting changes to the minimum number of lines of code to increase the chances of a consensual hard fork being accepted. The email threads discussed finding the best approach to solving the Segwit block size discount issue while balancing scalability and reliability. There were also discussions about Bitcoin's gradual improvements, which were simple enough for anyone with technical knowledge to understand at the beginning but now require a full website to explain.Sergio Demian Lerner proposed a soft-fork to 50% now and another soft-fork again to 75% discount later if required, while Johnson Lau suggested using 75% in a softfork and changing it to a different value or redefining the definition of weight with a hardfork later. Despite disagreements, Lerner clarified that he was mediating the discussion, and the email included some lighthearted banter between the participants.
