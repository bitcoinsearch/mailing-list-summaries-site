In various email exchanges between Bitcoin experts, the focus is on minimizing data transfer in cryptocurrency mining. One suggestion is to use forward error correction (FEC) for low data transfer and negligible probability of needing a round-trip. However, FEC schemes are complex and add additional metadata overhead. Set reconciliation is considered near optimal, and Yaron's non-interactive scheme is praised for its ability to immediately figure out the delta without further back and forth communication, each requiring an RTT.The conversation also includes discussions on creating a mining-only network for faster block propagation and incentivizing ahead-of-block transaction propagation. The idea of mempool synchronization brings up the issue of transaction expiration since lack of mempool syncing is currently the mechanism for tx expiry. Furthermore, there are discussions about improving the new-block process by using sparseblocks and ultra-fast block validation (UFBV) to make the new-block process more efficient for miners. Several email exchanges between Bitcoin developers and open source evangelists have been discussing various proposals to improve the efficiency and speed of block propagation in the network. One suggestion is to create a new mining-only network that could allow for insanely fast propagation of newly solved blocks. Another proposal involves using sparseblocks and ultra-fast block validation (UFBV) to improve the new-block process, but this is only useful when transactions are seen in advance. Additionally, there has been discussion about implementing set reconciliation techniques to reduce communication complexity between peers and exchanging mempool priority policies to achieve flexibility in which transactions to remember.In an email conversation on July 18, 2014, Bitcoin developer Kaz Wesley proposed a method for peers to exchange mempool priority policies and assign unique sequential IDs to transactions. Sparse block, invack message, and gettx were suggested as possible new messages. Ordering policies should allow arbitrary algebraic combinations of their parameters, thresholds, and Boolean combinations of sub-policies. However, this could lead to a completely heterogeneous mempool policy environment. A parallel mechanism for relay policies may be added to differentiate between what a node would keep in its mempool versus what it wouldn't even relay. Another idea was broadcasted regarding saving bandwidth by letting peers create a very good approximation of a block with just transaction selection policy information. Peers should exchange mempool priority policies suggested Wesley in another email. The idea is to assign unique sequential IDs for each transaction. Possible new messages include sparse block, invack message, and gettx.
